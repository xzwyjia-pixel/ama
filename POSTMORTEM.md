# RuleGuard / ZhenAudit 甄查 — 归档记录

**项目周期:** 2026-05 ~ 2026-06 (约 1 个月)
**存档日期:** 2026-06-03
**状态:** 主动放弃

---

## 做了什么

一个面向抖音直播间的 AI 合规审查工具，核心技术是"三方 Agent 辩论 + 裁判官收敛"的多模型协作流水线。

| 组件 | 文件 | 完成度 |
|---|---|---|
| 辩论引擎 | `ama/core/debate.py` | 可用，17 次真实辩论记录 |
| 商业化 API | `ruleguard_server.py` | 可用，FastAPI + IP/Cookie 限额 |
| 管理后台 | `public/admin.html` | HTML 骨架 |
| 违规词统计 | `scripts/violation_stats.py` | 可用，导出 Obsidian frontmatter |
| 前端页面 | `public/zhenaudit.html` | 可用，SPA 审查界面 |
| 配额管理 | `ruleguard_quota.py` | 可用，日 3 次免费 |

---

## 为什么放弃

**一句话: 市场不需要另一个合规预审工具。** 抖音官方审核 + 现有竞品已覆盖 80% 需求，剩余长尾养不活一个 SaaS。

具体判断:

1. **护城河为零。** regex 违规词库 + DeepSeek API 调用的组合，任何懂 Prompt Engineering 的人一周可复现。没有独有数据、没有平台关系、没有网络效应。

2. **客户不付费。** 合规预审是"保险"型需求——不出事不觉得需要，出事才知道疼。中小商家不会为"可能被罚款"这件事每月付 99 块。目标客户画像模糊。

3. **依赖 DeepSeek，成本不可控。** 每次审查调 3+1 个 Agent（3 轮辩论 + 1 次裁判官收敛），Token 消耗大。规模化后毛利极薄，除非自建小模型——那又是另一条路。

4. **抖音官方风险。** 如果抖音升级合规检测能力（或开放官方 API），第三方工具直接归零。

5. **不是什么都能卖。** 这个项目的核心资产是"对抖音违规规则的理解"，这是一门咨询服务，不是软件产品。试图把它包装成 SaaS 是方向性错误。

## 核心教训

**先验证付费意愿，再写代码。** 这个项目的构建顺序完全反了:

- 正确: 找 10 个 MCN → 免费试用 → 问愿不愿付费 → 愿意再建产品
- 实际: 建完辩论引擎 → 建完 API → 建完前端 → 开始想怎么卖 → 发现没人买

**数据护城河 > 算法护城河。** 再精巧的 Prompt 模板也不如 10 万条标注数据值钱。应该先积累数据（哪怕是手工标注），而不是先优化辩论协议。

**B 端 SaaS 不适合 solo maker。** 企业合规工具需要销售团队、合同、法务、SLA——不是一个人能搞定的。Solo 更适合做开发者工具或内容型产品。

## 哪些可以复用

| 资产 | 可复用于 |
|---|---|
| `ama/core/debate.py` 多 Agent 辩论模式 | 任何需要"多角度 + 强制收敛"的场景（代码审查、合同审查、学术仲裁） |
| `ruleguard_quota.py` 双层限额 | 任何需要 IP+Cookie 免费额度的 API 产品 |
| `violation_stats.py` → Obsidian 管线 | 任何需要定时生成分析报告的项目 |
| 违规关键词库 (7 大类, 30+ 正则) | 单独的合规类内容产品（如公众号工具文） |

## 如果重来一次

1. **不做辩论引擎，先做违规词知识库。** 卖数据，不卖工具。
2. **用 Notion/飞书文档做 MVP，不用代码。** 手动审查 100 条话术，看能不能总结出 enough 有价值的模式。
3. **先找到 3 个愿意付费的客户。** 没有 3 个，不写一行代码。
